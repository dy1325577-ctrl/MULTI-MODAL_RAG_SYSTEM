Multimodal RAG — System Architecture

A PDF of text, tables, and images becomes two searchable spaces. A user asks with text and an image; four retrieval paths converge, and a 4-bit vision model answers from what they hit.

Storage — the shared hub

Persistent on disk

Phase 1 writes here; Phase 2 reads from here. Two vector spaces plus a raw-payload store. Metadata is stored as fields, never embedded — that is what keeps the "images + tables + metadata" problem from tangling the embedding space.

PHASE 2

Query

Online · per message

The user sends text and (optionally) an image from the dashboard. It fans into four retrieval paths that hit the two indexes, the results are fused, and the raw payloads are pulled back for the answer.

User input — text + image
Streamlit · st.chat_input + st.file_uploader
↓ fans into four retrieval paths ↓
1 query text bge Index A
2 query text CLIP text tower Index B
3 query image CLIP image tower Index B ★ Semantic image match — finds the PDF image even when the query image is only similar, not identical.
4 query image VLM caption bge Index A ★ Second chance via text — describes the image, then matches its meaning against summaries.

★

Why the "similar image" requirement is met: paths 3 and 4 both start from the user's image. CLIP similarity is semantic rather than pixel-exact, so a redrawn or screenshotted chart still lands near the original — and the VLM-caption path catches whatever CLIP misses.
